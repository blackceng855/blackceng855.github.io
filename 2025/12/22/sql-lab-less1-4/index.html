进入sql注入世界的第一天

2025/12/22 从零开始的sql注入生活 sql_lab

union型注入（less1-less4)

原理

其实刚开始做这类型题时我一直在想为什么

?id=1’ 错误

?id=1’–+ 回显

后来才发现我想的太过于复杂，只是相当于一个变量的替换

SELECT * FROM users WHERE id='1'

由于他get的是?id=后的值，所以当我输入1’–+时，返回去的实际上是SELECT * FROM users WHERE id=’1’–+’ ，从而使前面正常输出的同时给我们后续构造恶意语句留足了空间接着又可以注释掉我们输入的‘从而保证语句的正常执行。

接下来的2，3，4题都是同样的原理，只是他们所用的封闭符号不一样罢了，只要试出来封闭符号就可以解决了例如’,’),”),”,’)),”))。

判断注入点漏洞类型

开始我们发现存在SQL注入之后，下一步操作就是要判断注入点的漏洞类型。究竟是字符型还是数字型。

  1. 数字型：直接将注入点的数据当做数字处理，带入了查询语句中，不需要符号闭合。

  2. 字符型：在拼接SQL查询语句时，对用户的输入使用了单引号或双引号进行闭合。需要进行闭合（强调一点：这和数据库中字段本身的数据类型无关，关键在于SQL语句拼接时是否有引号闭合）

    判断方法

    ?id=1 and 1=1

    正常回显

    ?id=1 and 1=2

    当1=1返回了数据，1=2返回不了数据是数字型注入，

    因为这证明我们恶意输入的SQL语句直接带入了后端查询语句中(SELECT * FROM users WHERE id=1 and 1=2 LIMIT 0,1)，说明输入点没有引号包裹。

    当1 and 1=2仍然正常回显，则可能是字符型（需要闭合引号）或逻辑始终为真。

    ?id=1
    正常回显

    ?id=1'
    如果报错是字符型。
    因为单引号破坏了语句原来的闭合结构，证明需要闭合，

    如果没有报错是数字型。
    因为输入 1’ 通常 不会导致 SQL 语法错误。数据库引擎会尝试将 1’ 转换为数字。大多数数据库会

    要么 忽略 单引号及其后面的非数字字符，只取 1（可能伴随一个隐式转换警告，但查询语法正确，还是能执行成功）。

    要么 报“数据类型转换错误” (如 Conversion failed when converting the varchar value ‘1’’ to data type int)，但这 不同于 字符型注入中 ‘1’’ 导致的 语法错误 (如 Unclosed quotation mark after the character string)。关键区别在于错误类型（转换错误 vs 语法错误）和是否破坏了语句结构。

    因此只有字符型需要进行闭合。

列数查询

构造完基本形式，使用order by语句查询一共有多少列。

?id=1' order by 3 --+

ORDER BY 3 表示按查询结果中的第3列进行排序。如果查询结果只有2列，那么 ORDER BY 3 就会引用一个不存在的列，从而导致数据库报错。 通过递增 ORDER BY 后的数字（如1,2,3,4…）直到报错，即可确定查询返回的列数（报错前的那个数字就是列数）。–+ 注释掉原语句中可能存在的 LIMIT 等后续部分，保证 ORDER BY 能正确执行和触发错误。

为什么要判断字段数：在接下来的联合查询形成的语句会是select * where id = 1 union select 1,2,3，联合查询两个select的列数必须一致，*代表所有列，因此要先知道总列数，才能正确设置联合查询的列数

判断回显

当我们判断完列数后,由于不是所有列的数据都要进行返回的,所以我们需要查看回显位,判断哪些位置回显的有数据.我们使用特殊符号来标记一下,如果返回了特殊符号,说明这个地方是可以回显的.

  • /?id=1' union select 1,2,3--+ ‘数据回显，但是没有回显注入点
    /?id=0' union select 1,2,3--+ ‘回显注入点为2和3，知道回显点位，我们就可以在该回显点处构造语句来回显我们想要的数据。

  • 为什么第二行修改为id=0成功回显：在刚刚的白盒中，我并未介绍LIMIT 0,1的作用，其作用为在查询到的所有数据中，从第0行开始，输出1行。当id=1时，是能够正常查询到数据的，于是只有这一行被输出了。因此我们将id设置为0（或者负数），使得主查询查不到数据，后面联合查询的数据就能正常输出了。主查询没有数据，自然就只会输出后面数据了。

  • /?id=0' union select 1,2,database()--+ '数据库security

    /?id=0' union select 1,group_concat(table_name),3 from information_schema.tables where table_schema='security' --+ ‘表名emails,referers,uagents,users

    /?id=0' union select 1,group_concat(column_name),3 from information_schema.columns where table_name='users' and table_schema='security' --+ ‘字段id,username,password

    /?id=0' union select 1,group_concat(username),group_concat(password) from security.users --+

    • database()：当前数据库名，代替3的位置后，数据库名就会被输出。
    • group_concat：将得到的所有结果拼接起来并返回，如果不拼接，只会显示查到的第一行结果。
    • information_schema：是mysql自带的一个特殊的数据库，它记录了所有的表名、列名，通过它，我们就能从数据库名一层层的拆解出我们想要数据的列名、表名，并查询到想要的数据。
    • 最后就能拿到所有的用户名和密码了。
